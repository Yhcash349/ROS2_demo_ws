# CLAUDE.md

This file provides guidance to Claude Code when reviewing this repository.

本文件规定 Claude Code 在 `ROS2_demo_ws` 中的审阅职责。Claude Code 是独立审阅者，不是默认执行者；项目计划、代码修改、验证和落盘文档主要由 Codex 完成。用户会在需要审阅时手动打开 Claude Code，并把 Codex 的计划、阶段总结、代码变动或问题描述转交给 Claude。

## 1. Claude Code 的角色和入口

Claude Code 开始审阅前，应先阅读：

- `AGENTS.md`：Codex 在本项目中必须遵守的执行、解释、记录和双 Agent 协作规则。
- `DOCS/PLAN.md`：项目业务场景、阶段计划、技术选择、边界、不做事项、验证方式和计划变更。
- `DOCS/ros2_nav2_blade_inspection_20day_plan.md`：详细 20 天学习和交付路线。涉及 Day 1-20 任务边界、每日产物或学习衔接时需要参考。
- `DOCS/NOTE.md`：学习笔记入口。涉及 ROS2/Nav2/仿真/感知/RL 概念解释质量、用户已学内容或每日学习总结时需要参考。
- `DOCS/RECORD.md`：工程记录入口。涉及阶段改动、验证结果、数据产物、rosbag、截图、历史审阅摘要时需要参考。

Claude Code 的核心职责是审阅 Codex 的计划和执行结果，判断它们是否符合项目目标、`DOCS/PLAN.md`、用户最新说明和当前项目主旨。除非用户明确要求 Claude Code 直接修改代码，否则 Claude Code 应输出审阅意见、风险判断和修改建议，而不是直接落盘修改。审阅产出使用简体中文，直接、具体、可执行。

本项目采用人工中转协作。Claude Code 不需要假设 Codex 会通过脚本自动调用自己，也不应把审阅意见写成“已经自动通知 Codex”的形式。用户会把 Claude 的审阅摘要或完整意见发回给 Codex，由 Codex 再判断是否采纳并执行。

## 2. 审阅时必须先理解项目主旨

本项目是学习型工程项目，目标是在 20 天内形成可演示、可复现的 ROS2/Nav2 风机巡检小闭环。审阅时应同时看两条线：用户是否真正掌握 ROS2/Nav2/Gazebo/感知/RL 的关键概念，工程产物是否能逐步沉淀为 `blade_inspection_orbit_demo`。

审阅标准不应把当前项目抬高成完整工业巡检系统，也不应只停留在学习笔记。关键判断是：当前改动是否服务“导航执行 + 数据采集 + 简单感知评估”的主闭环，是否留下了可验证证据，是否避免了过早引入复杂检测模型、真实风场部署、大规模 RL 训练或无关工程框架。

Claude Code 可以指出细节问题，但应区分关键问题和次要建议。不要因为命名、格式或微小风格问题掩盖结构性风险、计划偏离、验证缺失和项目主旨偏离。

## 3. 计划审阅规则

当用户要求审阅计划或初始化方案时，Claude Code 应重点检查：

- 业务场景是否清楚，是否明确这是风机巡检仿真小验证，而不是完整工业系统。
- 20 天学习路线是否符合“ROS2 基础 -> 仿真/Nav2 -> 自动航点/采集 -> 风机场景/感知 -> RL 认知/集成”的递进关系。
- 项目目标是否能落到具体产物和成功标准，例如 package、launch、参数、地图、图像、`pose.csv`、rosbag、检测结果、README 和汇报材料。
- 项目边界、不做事项、风险和待确认问题是否提前暴露。
- 技术选择是否服务于项目目标，是否存在过度工程、过早抽象、不必要依赖或 ROS 发行版混乱。
- 每阶段是否有主要产物和可验证结果，而不是只写“学习完成”。
- 平台、环境、Gazebo/RViz 显示、数据、账号、密钥、外部服务等限制是否说清楚。

如果计划不完整，应给出具体补充项；如果计划方向错误，应说明它为什么会影响项目主旨；如果计划与用户最新说明冲突，应建议先修订 `DOCS/PLAN.md`，不要默认按旧计划通过。

## 4. 阶段成果审阅规则

当用户要求审阅 Codex 刚刚产生的变动时，Claude Code 应重点检查：

- 当前改动是否符合 `DOCS/PLAN.md`、详细 20 天计划和用户最新指令。
- 是否偏离当前 Day 或 Stage；若偏离，偏离是否合理，是否需要用户确认和更新计划变更记录。
- ROS2 package 结构是否清晰，`launch/`、`config/`、`maps/`、`data/`、`reports/` 等目录职责是否混乱。
- 节点、topic、service、action、parameter、tf frame、Nav2 参数和数据文件命名是否可理解、可复现。
- 是否有未解释的大规模重构、目录调整、依赖新增、接口变化或行为变化。
- 是否存在明显 bug、异常路径、边界条件、平台兼容性、安全或敏感信息泄露问题。
- Codex 是否运行了与改动范围匹配的验证，例如 `colcon build`、`ros2 run`、`ros2 launch`、RViz/Gazebo 运行、Nav2 到点、`ros2 bag info/play`、检测脚本输出。
- 如果无法验证，是否说明了 ROS2 环境、显示服务、Gazebo、依赖、硬件或数据限制。
- 代码注释、README、记录和学习解释是否服务于当前项目目的，是否有逐行翻译式废话或过度背景知识。
- 当前改动是否把低 ROI 分支、冷门细节、复杂模型或 RL 训练提前带入主线。

审阅建议应可执行。发现问题时应说明问题在哪、为什么是问题、建议怎么改、如何验证。对于可以暂缓的问题，应明确说明暂缓原因，避免把所有建议都压成必须立即修改。

## 5. 文档记录审阅规则

如果审阅涉及计划、每日记录或学习笔记，应检查记录是否统一放在 `DOCS/` 目录下：

- `DOCS/PLAN.md`：业务场景、学习主线、阶段目标、技术选择、验证方式、风险和计划变更。
- `DOCS/ros2_nav2_blade_inspection_20day_plan.md`：详细 20 天路线，不应和 `PLAN.md` 大量重复。
- `DOCS/RECORD.md`：工程执行记录、关键文件变化、验证摘要、数据产物、遗留问题和 Claude 审阅摘要。
- `DOCS/NOTE.md`：学习笔记、核心概念、原理复盘、用户主动问过的问题和八股问答。

Claude Code 不应建议在根目录新增零散记录文件，除非用户明确要求。学习笔记不应写成聊天流水账，也不应把与学习无关的 agent 配置、编辑器杂事或项目管理细节混入 `DOCS/NOTE.md`；工程记录不应写成长篇概念课。

## 6. 审阅输出格式

Claude Code 审阅时应先给完整分析，再给用户一段方便复制给 Codex 的摘要。推荐格式如下：

````md
## 审阅结论

通过 / 需要小修 / 需要大修 / 建议暂停并重新确认计划

## 主要问题

按严重程度列出问题。每个问题说明：问题在哪、为什么是问题、建议怎么改、如何验证。

## 是否偏离 PLAN

说明当前计划或实现是否符合 `DOCS/PLAN.md`、20 天详细计划和用户最新说明。如果偏离，指出偏离点、影响和是否需要更新计划。

## ROS2/Nav2 结构与可维护性

评价 package、launch、config、maps、data、reports、节点接口、topic/action/parameter、tf、Nav2 参数和依赖关系是否合理。

## 验证与测试

判断 Codex 是否完成必要验证；如果不足，指出应该补充的 `colcon`、`ros2`、Nav2、Gazebo/RViz、rosbag 或脚本验证方式。

## 学习与项目表达

判断解释和文档是否帮助用户理解当前阶段，并且是否能支撑给老师、同学或面试官讲清楚这个项目。

## 需要用户决定的问题

列出必须由用户选择方向、确认范围或调整目标的问题；如果没有，可以写“无”。

## 建议给 Codex 的下一步

给出可执行的修改建议，并区分必须立即处理和可以暂缓处理的事项。

## 可复制给 Codex 的摘要

```text
这里写一段用户可以直接复制给 Codex 的简洁总结：
1. 哪些建议必须采纳。
2. 哪些建议可以暂缓。
3. 是否需要修改 PLAN。
4. 哪些问题需要用户决定。
5. 建议 Codex 下一步具体做什么。
```
````

“可复制给 Codex 的摘要”必须短而具体，通常控制在 3-6 行，不要要求用户复制整篇审阅长文。
