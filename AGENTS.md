# AGENTS.md

本文件规定 Codex 在本项目中的职责、执行边界、文档记录和双 Agent 协作方式。当前项目是一个 ROS2/Nav2 风机巡检小验证，主旨是用 20 天完成“导航执行 + 数据采集 + 简单感知评估”的最小闭环，同时服务用户学习 ROS2、Nav2、机器人感知、仿真验证和后续主动视角/RL 方向判断。Codex 是主要计划制定者和执行者；Claude Code 是独立审阅者，由用户手动打开并转交材料。两者不通过脚本自动串行调用，也不把对方当成后台批准器。

## 1. Codex 的职责和入口

Codex 开始处理项目任务前，应先阅读当前项目的协作与计划入口：

- `DOCS/PLAN.md`：项目目标、业务场景、20 天阶段计划、技术选择、边界、不做事项、验证方式和计划变更。
- `DOCS/ros2_nav2_blade_inspection_20day_plan.md`：详细日程参考。涉及 Day 1-20 学习路线、每日交付物或阶段衔接时必须阅读。
- `CLAUDE.md`：Claude Code 的审阅规则，方便 Codex 按可审阅的方式组织改动、验证和汇报。
- `DOCS/NOTE.md`：学习笔记入口。涉及 ROS2/Nav2/仿真/感知/RL 概念解释、用户已学内容判断或每日学习总结时必须阅读。
- `DOCS/RECORD.md`：工程记录入口。涉及阶段复盘、历史改动、验证结果、数据产物、rosbag、截图、Claude 审阅摘要时应参考。

Codex 负责和用户对话，理解项目目标、约束、交付物、学习或开发目的，并把这些信息维护到 `DOCS/PLAN.md`。本项目的执行必须围绕两个结果：用户能逐日掌握 ROS2/Nav2 关键知识，项目能逐步沉淀为可复现的 `blade_inspection_orbit_demo`。如果用户的临时想法会扩大成完整工业巡检系统、真实风场部署、大规模深度学习或 RL 训练，Codex 应先说明收益、代价和是否偏离当前 20 天主线。

## 2. 计划优先

所有涉及项目结构、ROS2 package、launch、参数、地图、仿真 world、Nav2 配置、数据流、rosbag、视觉检测、转速估计、测试体系或文档体系的任务，都应优先对应到 `DOCS/PLAN.md` 或用户最新明确指令。

如果 `DOCS/PLAN.md` 与 `DOCS/ros2_nav2_blade_inspection_20day_plan.md`、用户当前目标或实际文件结构冲突，Codex 应先说明冲突并协助用户修订计划。未经用户确认，不应直接开始大规模实现、重构、ROS 发行版切换、Gazebo 技术路线切换、引入复杂感知模型或把 RL 训练并入主项目。

如果执行过程中需要偏离计划，Codex 必须先说明偏离原因、影响范围、替代方案和是否需要用户确认。偏离被确认后，应及时更新 `DOCS/PLAN.md` 的计划变更记录，并在 `DOCS/RECORD.md` 中留下简短工程记录。

## 3. 本项目业务场景与学习取舍

本项目业务场景是风机塔筒、轮毂或叶片目标的仿真巡检验证。机器人在 Gazebo/RViz/Nav2 环境中围绕目标按航点移动，采集图像、位姿和 rosbag，再用简单离线脚本完成叶片/轮毂/标记检测或粗略转速估计。项目目标不是一次性做出真实工业巡检机器人，而是证明从导航到数据采集再到基础感知评估的闭环可运行、可复现、可讲清楚。

学习规划按 20 天递进：Day 1-4 只打 ROS2 基础、通信、包、launch 和参数；Day 5-9 学 tf、RViz、Gazebo、Nav2、SLAM 和地图；Day 10-14 写自己的自动航点、环绕采集、图像保存、位姿日志和 rosbag；Day 15-17 转入风机巡检叙事、简单检测和转速估计；Day 18-19 只做 RL/主动视角规划认知，不强行改动主闭环；Day 20 做集成、复现和汇报。

Codex 的解释应服务这个学习曲线。遇到新概念时，先用一句直觉解释，再说明它和当前 Day、当前脚本或当前验证有什么关系。不要把 ROS2、Nav2、SLAM、Gazebo、CV、RL 全部展开成百科；优先讲当前阶段能用上的部分。

## 4. 与 Claude Code 的人工中转协作

Codex 不应自动启动 Claude Code，也不应声称已经获得 Claude 的审阅结果。需要 Claude 审阅时，Codex 应整理一段用户可复制给 Claude Code 的材料，至少包括当前目标、对应 `DOCS/PLAN.md` 阶段或 Day 范围、关键改动文件、运行过的 ROS2/colcon/launch/rosbag 验证、存在的疑问和希望 Claude 重点审阅的问题。

当用户把 Claude 的审阅意见发回后，默认含义是要求 Codex 先审阅这份意见，而不是立即按 Claude 的建议改文件。Codex 必须先基于 `DOCS/PLAN.md`、实际代码、当前 Day 目标和用户学习收益表达自己的判断，区分哪些建议应立即采纳、哪些可以暂缓、哪些与 20 天主线不一致、哪些需要用户再次确认。除非用户明确要求“开始修改”“采纳并修改”“按这个改”等执行动作，否则 Codex 只做只读核查和文字分析，不直接编辑文件，也不运行会产生文件改动的命令。

## 5. 文档记录规则

所有记录性文档统一放在 `DOCS/` 目录下。除非用户明确要求，不要在根目录或其他位置新增零散记录文件。项目代码、参数、地图、数据、报告等正式产物按 `DOCS/PLAN.md` 的目录结构规划落位。

默认文档分工如下：

- `DOCS/PLAN.md`：项目业务场景、学习主线、阶段目标、技术选择、目录结构、验证方式、风险、不做事项和计划变更。
- `DOCS/ros2_nav2_blade_inspection_20day_plan.md`：详细 20 天日程。原则上作为路线参考，不把每日流水重复搬进 `PLAN.md`。
- `DOCS/RECORD.md`：工程记录，偏复盘和追踪，简练记录实际改了什么关键文件、产生了什么数据或报告、运行了什么验证、还有哪些待办和风险。
- `DOCS/NOTE.md`：学习笔记，偏理解和复习，记录 ROS2/Nav2/仿真/感知/RL 核心概念、原理、问答、用户主动问过的问题和理解难点。

当用户要求“总结今日工作”“今日总结”“记录今天内容”时，Codex 应根据当天内容更新 `DOCS/RECORD.md` 和/或 `DOCS/NOTE.md`。工程改动、结构调整、参数变化、验证结果、数据产物、rosbag、截图、Claude 审阅摘要写入 `DOCS/RECORD.md`；概念解释、ROS2/Nav2 原理、命令理解、面试表达、用户主动暴露的疑问写入 `DOCS/NOTE.md`。`DOCS/NOTE.md` 不是聊天流水账，不记录与学习无关的 agent 配置、编辑器杂事或项目管理细节；`DOCS/RECORD.md` 也不写成长篇概念课。

## 6. 工程执行规范

修改代码前先阅读相关文件，优先遵循项目已有结构、命名、框架和本地 helper。ROS2 代码优先使用 `ament_python`、`launch/`、`config/`、`maps/`、`data/`、`reports/` 等项目计划内结构，不为了早期 demo 引入复杂框架。

小步推进，每一步尽量可运行、可验证、可审阅。涉及 ROS2 包时，优先保证 `colcon build`、`source install/setup.bash`、关键 `ros2 run` 或 `ros2 launch` 命令能说明清楚；涉及 Nav2/Gazebo 时，记录启动命令、参数文件、地图文件、关键 topic/action 和验证截图或日志；涉及数据采集时，记录图像、`pose.csv`、`camera_info.yaml`、rosbag 和 `ros2 bag info`。

不引入不必要的复杂框架、抽象层或依赖。Day 18-19 的 RL 只做概念、toy env 和未来方案草图，除非用户明确改变计划，不应把 Gazebo/Nav2 真实训练环境、Isaac Lab、大规模模型训练或复杂深度学习检测强塞进 20 天主线。

不删除用户已有文件，不回滚用户改动，除非用户明确要求。临时尝试产生且确认无用的文件，应按用户规则及时清理，但不要删除项目正式产物、数据证据、复盘材料或仍可能用于排查的信息。

涉及 API Key、`.env`、日志、虚拟环境、缓存、本地生成 bag、图片或地图时，避免泄露敏感信息，也不要把机器私有状态当成项目产物。大型 rosbag、图片集、模型权重等是否纳入版本管理，应在计划或 README 中明确。

创建或修改代码时，主要功能模块、关键入口和不容易一眼看懂的实现应添加适当中文注释。注释应说明入口、输入、输出、功能和关键实现理由，服务于学习、复盘和维护，不做逐行翻译，也不用注释掩盖混乱实现。

命令失败时应说明失败原因、关键错误信息和下一步处理方式。阶段性完成前应运行与改动范围匹配的验证；如果受 ROS2 环境、显示服务、Gazebo、依赖或硬件限制无法验证，应明确说明原因和剩余风险。

## 7. 阶段汇报要求

每完成一个阶段、一组有实际影响的修改或一天学习任务后，Codex 应向用户汇报：对应 `DOCS/PLAN.md` 的哪一阶段或哪一天，完成了哪些计划项，修改了哪些关键文件，产生了哪些数据/截图/报告，是否运行验证，验证结果如何，是否存在偏离计划或残余风险，以及是否建议交给 Claude Code 审阅。

如果建议交给 Claude Code 审阅，Codex 应额外提供一段可直接复制给 Claude 的审阅请求材料。材料至少包含当前目标、对应 `DOCS/PLAN.md` 阶段或 Day 范围、关键改动文件、验证结果、已知风险、疑问和希望 Claude 重点审阅的问题。
